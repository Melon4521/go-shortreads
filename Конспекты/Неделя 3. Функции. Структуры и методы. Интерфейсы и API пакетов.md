# Функции

## Синтаксис

**Функции в Go** - переиспользуемые исполняемые блоки кода, также являются значениями.
Объявление функции включается в себя ее **сигнатуру** - параметры + тип возвращаемого значения (если есть).

```go
func greeting(name string, message string) {
	// name, message - параметры функции
	fmt.Printf("Привет, %s. %s", name, message)
}

greeting("Иван", "Как дела?") // Вывод: "Привет, Иван. Как дела?"
// "Иван", "Как дела?" - аргументы функции
```

Для каждого параметра функции мы указываем свой тип, в итоге объявляются локальные переменные с этими именами внутри функции. Если несколько подряд идущих параметров имеют одинаковый тип, то можно сгруппировать:

```go
func summary(a, b int) int { ... }
```

Если функция возвращает значение, то мы указыавем его тип. Но функция также может возвращать не одно значение, тогда мы указываем их типы в `()` через запятую: 

```go
func summaryAndMult(a, b int) (int, int) { ... }
```

В сигнатуре можно заранее указать имена переменных, которые нужно вернуть:

```go
func summaryAndMult(a, b int) (sum, mult int) {
	sum = a + b
	mult = a * b
	return // ничего не указываем
}

summaryAndMult(2, 5) // 7 10
```

> [!important]
> Функции в Go - это значения специального типа (объекты первого класса). Поэтому мы можем работать с ними как со значениями: присваивать в переменную, передавать как аргумент в другую функцию и возвращать ее.
> 
> ```go
> // анонимная функция (опускаем имя в объявлении)
> summary := func (a, b int) int { 
> 	return a + b
> }
> ```
> 
> ```go
> func returnFunc() func (int, int) int {
> 	return func (a, b int) int {
> 		return a + b
> 	}
> }
> ```
> 
> ```go
>func doSmt(fn func ()) { // fn - колбэк-функция
>	fn()
>}
>
>doSmt(func () { fmt.Println("Hello!") })
> ```

## Доступ к функция

В Go действует простой принцип - если название функции начинается с заглавной буквы, то такая функция доступна и вне текущего пакета, а если с маленькой буквы, то только внутри текущего пакета

```go
package main

import "fmt"

func main() {
	// В пакете fmt есть функция getField
	fmt.getField() // ошибка доступа
	fmt.Println() // тут все ок
}
```
## Замыкания

**Замыкания** - это функции, которые на этапе объявления запоминают внешние переменные

```go
func makeCounter(initial int) func () int {
	count := initial
	
	return func () int {
		// Эта функция объявлена внутри тела makeCounter
		// Она запомнила переменные count и initial (параметр)
		count++
		return count
	}
}

counter := makeCounter(1)

counter() // 2
counter() // 3
```

# Кастомные типы

## Кастомные типы

**Кастомный тип** - наш собственный тип данных, который создается на основе базового типа

```go
type OurString string
type OurInt int

var customString OurString
fmt.Printf("Тип: %T, значение: %#v", customString, customString)
// Тип: main.OurString, значение: ""
```

Пример с коротким объявлением переменной (приведение типов):

```go
customInt := OurInt(5) // не можем просто := 5
```


# Структуры

## Структуры

**Структура** - набор полей **поле: значение**

```go
type Person struct { // struct - это тип данных
	Name string // название: тип
	Age int
}
```

#### Инициализация структуры

**Вариант 1**. Структура с дефолтным значением

```go
var John Person
fmt.Printf("%#v \n", John) // main.Person{Name:"", Age:0} 

Alex := Person{}
fmt.Printf("%#v \n", Alex) // main.Person{Name:"", Age:0} 
```

**Вариант 2**. С указанием значений полей

```go
Peter := Person{
	Name: "Peter",
	Age: 30,
}
```

**Вариант 3**. Значения по порядку

```go
Peter := Person{"Peter", 30}
```

Здесь при указании значений нужно соблюдать порядок полей и нельзя пропускать промежуточные поля.

#### Работа с полями структуры

К полям структуры можно обратиться через `.`:

```go
John.Name = "John"
John.Age = 30
fmt.Println(John.Name, John.Age) // John 30
```

#### Указатель на структуру

Мы можем взять указатель прямо на объявлении структуры:

```go
pointerIvan := &Person{"Ivan", 20}

fmt.Println(pointerIvan) // &{Ivan 20}
```

В Go можно не разыменовывать указатель структуры явно, это синтаксический сахар:

```go
fmt.Println((*pointerIvan).Age) // 20, так делать необязательно
fmt.Println(pointerIvan.Age) // 20, авторазыменование
```

> [!important]
> В функции структуры по умолчанию передаются по значению (копируются), если надо изменить структуру в функции, то нужно передать указатель

#### Методы структур

**Метод** - это фукция, которая закрепляется за определенным типом или указателем на этот тип. Этим типом может быть структура или любой кастомный тип. Принадлежность метода к типу обозначается указанием **получателя (receiver)** на тип. Получатель может быть двух видов:
- **value-receiver** (t T) - получатель по значению
- **pointer-receiver** (t \*T) - получатель по указателю

```go
type T struct {}

func (t T) methodNameV() {} // t - получатель типа T по значению
func (t *T) methodNameP() {} // t - получатель типа T по указателю
```

Методы вызываются на самом значении типа:

```go
var valueT T

valueT.methodNameV()
(&valueT).methodNameP()
```


> [!important]
> Можно вызывать методы с value-receiver у указателей и, наоборот, методы с pointer-receiver у значений, в Go происходит автоматическая подстановка:
> ```go
> var valueT T
> pointerValueT := &valueT
> 
> pointerValueT.methodNameV() // (*pointerValueT).methodNameV()
> valueT.methodNameP() // (&valueT).methodNameP()
> ```

#### Получатель по значению

Метод с value-receiver получает копию значения и внутри мы будем работать именно с копией. Такие методы нужны, чтобы **использовать** значения полей, но **не менять** их.

```go
type Square struct {
	Side int
}

func (s Square) Perimetr() {
	fmt.Printf("Периметр: %v", s.Side * 4)
}

square := Square{4}
square.Perimetr() // Периметр: 16
```

#### Получатель по указателю

Метод с pointer-receiver получает указатель на значение. Такие методы нужны, чтобы **изменять** значения полей.

```go
type Square struct {
	Side int
}

func (s *Square) Scale(multiplier int) {
	s.Side *= multiplier
}

square := Square{4}
square.Scale(2) // {Side: 8}
```


> [!important]
> Если хотя бы один метод работает с получателем по указателю, то и у всех остальных методов стоит реализовать получатель по указателю. Так принято.

> [!important]
> В методах, так же как и в функциях, можно управлять доступом через название:
> - Начинается с заглавной - доступен вне пакета
> - Начинается с маленькой - доступен только внутри текущего пакета

#### Композиция структур, встраивание (embedding)

Значения полей структур в Go могут быть любыми, и структурами тоже. Это композиция структур:

```go
type Address struct {
    Street string
    City   string
}

type Person struct {
    Name    string
    Age     int
    Address Address // Поле Address типа Address
}

// person.Address.Street / City
```

В Go также есть отдельнный механизм **встраивания (embedding)**, который является частным случаем композиции структур и позволяет получать значения полей вложенной структуры напрямую из внешней:

```go
type Person struct {
	Name string
	Age int
}

type Builder struct {
	Person // встроили Person в Builder, название поля совпадает с типом структуры
	Experience int
}

builder := Builder{
	Person: Person{"Ivan", 30},
	Experience: 5,
}

fmt.Printf("Тип: %T, значение: %#v", builder, builder)
// Тип: main.Builder, значение: main.Builder{Person:main.Person{Name:"Ivan", Age:30}, Experience:5}

fmt.Println(builder.Person.Age) // 30
fmt.Println(builder.Age) // 30, доступен напрямую из структуры Builder
```


>[!important]
>При обращении к полям и методам структуры поиск начинается из корневой структуры и идет вглубь по вложенным структурам. Поэтому если во внешней структуре будет поле `Field` и во вложенной, то найдется именно поле внешней структуры - это **затенение** полей. Чтобы получить именно поле вложенной структуры, нужно указать полный путь: `outer.Inner.Field`, а не `outer.Field`.

>[!warning]
>Если у двух разных вложенных структур (на одиноковом уровне вложенности) будет поле с одним и тем же названием, то при попытке получить это поле напрямую из внешней структуры, мы получим ошибку, так как компилятор не понимает, поле какой вложенной структуры мы хотим получить. В таком случае нужно указать полный путь: не `outer.Field`, а `outer.FirstInner.Field`/`outer.SecondInner.Field`.




# Интерфейсы

## Синтаксис

**Интерфейс** - специальный тип в Go, который представляет из себя набор сигнатур методов.

```go
type Runner interface {
	Run() string
}
```

> [!important]
> Интерфейсы можно встраивать в другой интерфейс (embeding), чтобы не дублировать сигнатуры методов:
> ```go
> type Flyer interface {
> 	Fly() string
> }
> 
> type Swimmer interface {
> 	Swim() string
> }
> 
> type Ducker interface {
> 	Flyer
> 	Swimmer
> }
> ```

## Значение Интерфейсного типа

Все значения, которые рассматривались ранее, относятся к значениям конкретных типов. Значения интерфейсных типов устроены сложнее: внутри интерфейса хранится знание о  конкретном типе и знание о значении этого конкретного типа.

#### 3 варианта значения интерфейсного типа

```
                     nil                  ConcreteType               ConcreteType
                    /                    /                          /
interfaceValue = nil       interfaceValue             interfaceValue
                    \                    \                          \
					 nil				  nil                        ConcreteValue
```

**Вариант 1**. Не задан конкретный тип и его значение, **nil-интерфейс**

>[!important]
>Значение по умолчанию для интерфейсного типа - `nil`

```go
type Runner interface {
	Run() string
}

var runner Runner // nil-интерфейс

fmt.Printf("Тип: %T, значение: %#v", runner, runner)
// Тип: <nil>, значение: <nil>

if runner == nil {
	fmt.Println("Runner is nil") // условие выполняется, будет вывод
}
```

> [!important]
> Определенному интерфейсу соответствует только то значение, которое реализует указанные в нем методы, то есть **имплементирует** интерфейс. Вот так нельзя:
> ```go
> runner = int64(1) // int64 does not implement Runner (missing method Run)
> ```
> Причем имплементация в Go происходит неявно: достаточно, чтобы у значения были реализованы все указанные методы и он автоматически удовлетворяет интерфейсу


**Вариант 2**. Задан конкретный тип, но без значения (т.е `nil`)

```go
type Human struct {
	Name string
}  

// структура Human имплементирует интерфейс Runner
func (h Human) Run() string {
	return fmt.Sprintf("%v бежит!", h.Name)
}

// в примере используем указатель, т.к его дефолтное значение nil 

var unnamedRunner *Human // указатель на имплементирующий тип Human

var runner Runner = unnamedRunner
fmt.Printf("Тип: %T, значение: %#v \n", runner, runner)
// Тип: *main.Human, значение: (*main.Human)(nil) 
```

Важный момент, в таком случае в интерфейсном значении уже содержится информация о типе Human, поэтому это уже не nil-интерфейс:

```go
if runner == nil {
	fmt.Println("Runner is nil") // условие не выполняется
}
```

**Вариант 3**. Задан конкретный тип и значение

```go
var namedRunner *Human = &Human{"Ivan"}

var runner Runner = namedRunner
fmt.Printf("Тип: %T, значение: %#v \n", runner, runner)
// Тип: *main.Human, значение: &main.Human{Name:"Ivan"}
```


> [!warning]
> Важно! Вызвать метод у значения интерфейсного типа можно только в том случае, если это полноценный интерфейс: задано конкретный тип и его значение
> ```go
> type Runner interface {
> 	Run() string
> }
> 
> var runner Runner
> 
> runner.Run() // Ошибка
> ```

## Пустой интерфейс

```go
var empty interface{} = "Hello!"
// или:
// var empty any = "Hello!"

fmt.Printf("Тип: %T, значение: %#v \n", empty, empty)
// Тип: string, значение: "Hello!"
```

Пустой интерфейс не содержит в себе сигнатуры. По сути, чтобы имплементировать его, не нужно реализовывать никакие методы, поэтому любое значение удовлетворяет пустому интерфейсу

## Полиморфизм: для чего вообще нужны значения интерфейсного типа?

**Полиморфизм** - это способность **разных** типов уметь выполнять одинаково обозначенное действие (метод интерфейса), которое реализованно по-разному.

```go
type Human struct {
	Name string
}  

// структура Human имплементирует интерфейс Runner
func (h Human) Run() string {
	return fmt.Sprintf("Человек %v бежит!", h.Name)
}

type Duck struct {
	Name string
	Surname string
}  

// структура Duck имплементирует интерфейс Runner
func (d Duck) Run() string {
	return fmt.Sprintf("Утка %v бежит!", d.Name)
}

// принимает любой тип, имплементирующий интерфейс Runner
func polymorphism(r Runner) {
	fmt.Println( r.Run() )
}

human := Human{"Ivan"}
duck := Duck{"Donald", "Duck"}

polymorphism(human) // Человек Ivan бежит!
polymorphism(duck) // Утка Donald бежит!
```

## Type Assertion

Из значения интерфейсного типа можно получить значение конкретного типа `T` с помощью следующей конструкции: `val, ok := interfaceVal.(T)`. Если конкретный тип в интерфейсе является типов `T`, то в `val` присвоится значение этого типа из интерфейсного значения и в `ok` будет true:

```go
func typeAssertion(r Runner) {
  if human, ok := r.(Human); ok {
    fmt.Printf("Человек! Тип: %T, значение: %#v \n", human, human)
  } else if duck, ok := r.(Duck); ok {
    fmt.Printf("Утка! Тип: %T, значение: %#v \n", duck, duck)
  } else {
    fmt.Printf("Неизвестный тип: %T\n", val)
  }
}

human := Human{"Ivan"}
duck := Duck{"Donald", "Duck"}
  
typeAssertion(human)
// Человек! Тип: main.Human, значение: main.Human{Name:"Ivan"}

typeAssertion(duck)
// Утка! Тип: main.Duck, значение: main.Duck{Name:"Donald", Surname:"Duck"}
```

## Type Switch

Более короткий способ проверок:

```go
func typeSwitch(r Runner) {
    switch val := r.(type) {
    case Human:
        fmt.Printf("Человек! Тип: %T, значение: %#v \n", val, val)
    case Duck:
        fmt.Printf("Утка! Тип: %T, значение: %#v \n", val, val)
    default:
        fmt.Printf("Неизвестный тип: %T\n", val)
    }
}
```

## Method set

У каждого типа есть понятие набора методов, которые он реализует. По этому набору определяется, имплементирует ли тип интерфейсу. И тут есть важный нюанс:
- Набор методов указателя на тип `*T` содержит все методы, реализованные у типа: и с получателем по значению `(t T)` и с получателем по указателю `(t *T)`
- Набор методов значения типа `T` содержит только методы с получателем по значению `(t T)`

```go
type Document struct {
    title string
    content string
}

// Метод с получателем по значению
func (d Document) GetTitle() string {
    return d.title
}

// Метод с получателем по указателю
func (d *Document) SetTitle(title string) {
    d.title = title
}

// Интерфейс только с методом по значению
type Reader interface {
    GetTitle() string
}

// Интерфейс с методами по значению и указателю
type Writer interface {
    GetTitle() string
    SetTitle(string)
}

doc := Document{title: "Заголовок", content: "Содержимое"}
// Document имплементирует только интерфейс Reader, но не Writer

pointerDoc := &Document{title: "Заголовок", content: "Содержимое"}
// *Document имплементирует интерфейсы Reader и Writer
```
