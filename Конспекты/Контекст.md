## Определение

**Контекст** - инструмент, который помогает управлять жизненным циклом горутин

**Контекст позволяет выполнять 3 полезные вещи:**
- Передавать сигналы отмены между горутинами, для остановки их выполнения
- Устанавливать ограничение по времени выполнения кода
- Передавать метаданные сквозь "слои" программы

В стандартном пакете `context` контекст является интерфейсом:

```go
type Context interface {
    Deadline() (deadline time.Time, ok bool) // Возвращает конкретное время, в которое контекст должен быть отменён, и флаг, который говорит, установлен ли дедлайн или нет
    Done() <-chan struct{} // Возвращает канал, который закрывается при отмене контекста
    Err() error // Возвращает ошибку как причину отмены контекста
    Value(key any) any // Позволяет получить значение из контекста по ключу
}
```

## Типы контекстов

В Go контексты работают по принципу дерева: есть корневые контексты, от которых отслаиваются дочерние контексты. Если корневой контекст отменяется, то и все его дочерние контексты также будут отменены. Когда контекст отменяется, канал `ctx.Done()` закрывается.

Мы можем создать пустой (корневой) контекст двумя способами. Эти методы возвращают нам структуры, реализующие интерфейс `context.Context`:

- `ctx := context.Background()` - пустой, начальный контекст, на основе которого мы дальше будем строить цепочку контекстов
- `ctx := context.TODO()` - пустой контекст-заглушка, который используется в коде, если на этом месте в будущем должен быть сделан работоспособный контекст. Не использовать в проде!

Управляющие типы контекстов. Все они создаются на базе родительского контекста и возвращают новый канал `ctx` и функцию отмены `cancel`:

- `context.WithCancel(parentContext)` - предназначен для отмены "вручную" через `cancel()`.

> [!warning]
> Для таких каналов с отменой важно всегда вызывать `cancel()` (зачастую через `defer`), чтобы обеспечить гарантию завершения контекста.

- `context.WithTimeout(parentContext, 5 * time.Second)` - позволяет установить время выполнения, после чего будет автоматически отменен. В примере указано 5 секунд. Мы все равно можем отменить его досрочно через `cancel()`.
- `context.WithDeadline(parentContext, time.Now().Add(5 * time.Second))` - почти как контекст с тайм-аутом, но вместо длительности, мы указываем конкретный момент во времени, когда он должен быть отменен. Мы все равно можем отменить его досрочно через `cancel()`.

Также есть варианты этих методов с постфиксом `Cause`, который позволяет передать причину отмены в виде ошибки в функцию `cancel`. После отмены получить причину можно, передав этот контекст в вызов `context.Cause(ctx)`:

```go
ctx, cancel := context.WithCancelCause(context.Background())

go func() {
	time.Sleep(200 * time.Millisecond)
	cancel(errors.New("Client closed browser tab"))
}()

<-ctx.Done() // блокируем main-горутину, чтобы дождаться отмены ctx ⬆️⬆️⬆️
fmt.Println("Err:", ctx.Err())            // context canceled
fmt.Println("Cause:", context.Cause(ctx)) // Сlient closed browser tab
```

>[!important]
>Если нам нужно создать такой дочерний контекст, который не отменится вместе с родительским, то используем `context.WithoutCancel(parentContext)`

>[!tip]
>Можно выполнить колбек-функцию сразу после отмены контекст с помощью `context.AfterFunc(ctx, func() {})`

## Передача значений в контексте

Можно положить некоторые метаданные (информацию по аутентификации и прочие сопровождающие запрос данные, но не параметры для выполнения бизнес-логики!) в контекст по ключу: `ctx := context.WithValue(parentContext, key, value)`

Получить значение из контекста по ключу можно с методом `Value`. Но так как он принимает и возвращает любой тип, то необходимо использовать **type assertion**:

```go
if id, ok := ctx.Value(requestIDKey).(string); ok {
	fmt.Println(id)
} else {
	log.Fatal("fatal!!!")
}
```

## Пример работы с контекстом

```go
func worker(ctx context.Context, id int) {
    for {
        select {
        case <-ctx.Done():
            fmt.Printf("Worker %d: остановка, причина: %v\n", id, ctx.Err())
            return
        default:
            fmt.Printf("Worker %d: работает...\n", id)
            time.Sleep(500 * time.Millisecond)
        }
    }
}

func main() {
    ctx, cancel := context.WithCancel(context.Background())
    
    // Запускаем несколько горутин
    for i := 1; i <= 3; i++ {
        go worker(ctx, i)
    }
    
    // Даем им поработать две секунды
    time.Sleep(2 * time.Second)
    
    // Отменяем контекст
    fmt.Println("main: отправляем сигнал отмены...")
    cancel()
    
    // Даём время на завершение
    time.Sleep(1 * time.Second)
    fmt.Println("main: программа завершена")
}
```

```bash
Worker 2: работает...
Worker 1: работает...
Worker 3: работает...
Worker 1: работает...
Worker 3: работает...
Worker 2: работает...
Worker 3: работает...
Worker 1: работает...
Worker 2: работает...
Worker 1: работает...
Worker 3: работает...
Worker 2: работает...
main: отправляем сигнал отмены...
Worker 3: остановка, причина: context canceled
Worker 2: остановка, причина: context canceled
Worker 1: остановка, причина: context canceled
main: программа завершена
```

1. На основе базового контекста `Background` создаётся контекст с явной отменой.
2. Далее в цикле запускаются три горутины `worker`, в которые происходит передача контекста для управления их жизненным циклом.
3. Внутри горутин-воркеров в бесконечном цикле происходит прослушивание канала `ctx.Done()` у контекста. Сначала `select` пытается понять, закрыт ли канал через `case`. Если контекст ещё не отменен, то чтение из канала блокирует `select` и выполняется блок `default` и так до того момента, пока не последует сигнал от контекста.
4. Код некоторое время ждёт, имитируя деятельность воркеров, а затем явно вызывает `cancel`, после чего все воркеры получают сигнал о том, что контекст завершён через канал `ctx.Done()` и код ещё даёт немного времени горутинам на завершение.

## Идиоматика при работе с контекстом

1. В функции контекст обязан стоять первым параметром
2. Контекст не надо хранить в структурах
3. В долгих операциях надо проверять контекст через чтение из канала `ctx.Done()` в `select` с неблокирующим вызовом `default`
4. Нужно корректно задавать тайм-ауты: неправильно, если дочерний контекст имеет больший тайм-аут, чем родительский
5. Не использовать `context.TODO` в рабочем коде
6. `context.Background` создается только на верхнем уровне контекстов
