## Определение

**Примитивы синхронизации** - способ синхронизировать работу горутин, чтобы сделать их выполнение более предсказуемым, избежать гонок состояний и данных. Каналы являются одним из примитивов синхронизации

Инструменты синхронизации лежат в пакетах `sync` и в `sync/atomic`

## `sync/atomic`

Многие инструменты из `sync`, о которых будет ниже, реализованы с помощью `atomic`. Пакет `sync/atomic` предоставляет низкоуровневые функции для работы с данными на уровне инструкций процессора.

Операции выполняются атомарно (неделимо). Это значит, что в процессе изменения значения ни одна другая горутина не получит доступ к данным. Это позволяет избежать **data race**. Методы пакета: `AddT`, `LoadT` и другие (`T` - какой-то тип).

## `sync.Mutex`

**Мьютекс** позволяет выдать эсклюзивный доступ к участку коду / данным для одной горутины. Когда первая горутина вызывает `Lock`, то она получает доступ к работе с кодом ниже вплоть до вызова `Unlock`. Когда другие параллельные горутины в этот же момент дойдут до вызова `Lock`, то они встанут в ожидание, так как **мьютекс** уже занят. Когда первая дойдет до вызова `Unlock`, то теперь другая горутина сможет заблокировать для себя участок кода. Этот подход позволяет исключить **data race**.

```go
var (
    mu    sync.Mutex
    count int
)

func increment() {
    mu.Lock() // блокируем
    
    count++ // безопасно меняем общую переменную
    
    mu.Unlock() // разблокируем
}
```

Также есть `RWMutex`, помимо обычных `Lock` и `Unlock`, он имеет методы `RLock` и `RUnlock` для защиты читающего кода. Такой **мьютекс** ускоряет работу, когда горутин-читателей много, а пишущих - мало. Обычный **мьютекс** в такой ситуации создал бы большую очередь чтения. Но `RWMutex` разрешает параллелльное выполнение читающего кода и не создает очередность, если в этот момент никакая горутина не записывает. Как только в выполнении появляется горутина с обычным `Lock`, она встает в очередь и ждет (в это время новые читатели становятся за ней в очередь), пока текущие читатели закончат работу (`RUnlock`), после получает экслюзивный доступ: все читающие блокируются в ожидании.

## `sync.WaitGroup`

**WaitGroup** позволяет ожидать выполнение какого-то определенного количества горутин без преждевременного завершения. Это структура, которая имеет следующие методы:

- `wg.Add(n)` - добавляет в счетчик ожидания горутин 
- `wg.Done()` - вызывается в конце работы одной горутины, уменьшает счетчик ожидания
- `wg.Wait()` - блокирует выполнение программы до окончания выполнения всех ожидаемых горутин (пока внутренний счетчик не обнулится)

```go
func main() {
    var wg sync.WaitGroup
    
    wg.Add(2)
    
    for range 2 {
        go func() {
            defer wg.Done() // принято вызывать через defer
            fmt.Println("Hello World")
        }()
    }
    
    wg.Wait()
}
```

В данном примере 2 горутины с `Println` будут выполнены, т.к `wg.Wait()` блокирует завершение горутины `main`

## `sync.Once`

Структура с методом `Do`, который принимает колбэк-функцию и гарантирует, что она выполнится один раз первой горутиной, а остальные горутины не смогут ее вызвать.

```go
func initialize() {
    fmt.Println("Инициализация ресурсов (выполняется один раз)...")
}

func main() {
    var wg sync.WaitGroup
	var once sync.Once
    
    // запускаем 10 горутин, которые хотят выполнить инициализацию
    for i := 0; i < 10; i++ {
        wg.Add(1)
        
        go func(id int) {
            defer wg.Done()
			
            once.Do(initialize) // выполнится 1 раз
            fmt.Printf("Горутина %d завершила работу\n", id)
        }(i)
    }
    
    wg.Wait()
    fmt.Println("Программа завершена.")
}
```
