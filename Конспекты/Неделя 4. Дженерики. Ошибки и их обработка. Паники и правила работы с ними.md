# Дженерики

## Синтаксис

**Дженерики** - способ сделать код чище и избавиться от вариаций одного и того же кода для разных типов данных. С их помощью мы можем писать функции, которые могут принимать разные типы

В объявлении функции появляется новый блок **type parameters**: после имени функции в `[]` указываем название для **параметра типа** и после указываем **ограничения типа**, которым будет удовлетворять **параметр типа**:

```go
// функция работает или с int64 или с float64
func sum[T int64 | float64](numbers []T) T {
	var sum T
	
	for _, num := range numbers {
		sum += num
	}
	
	return sum
}

ints := []int64{1, 2, 3}
floats := []float64{0.5, 0.2, 0.3}

fmt.Println(sum(ints)) // 6
fmt.Println(sum(floats)) // 1.0
```

## Ограничения типа

На самом деле в блоке **type parameters** ограничение типа является интерфейсом. Мы можем передать в качестве ограничения заранее объявленный интерфейс:

```go
// аналог примеры выше

type Numbers64 interface {
	int64 | float64
} 

func sum[T Numbers64](numbers []T) T { ... }
```

> [!important]
> Раньше интерфейсы были только как набор методов, сейчас это может быть и набором типов. Для ограничения типа в дженериках нужно использовать именно интерфейсы как набор типов (либо встроенные, о которых ниже)

Теперь про сами варианты ограничения типов.

**Вариант 1**. Интерфейс с набором типов

```go
[T int64] // только int64
[T int | int32 | int64] // Объединение, "один из"
[T ~int] // Для кастомных типов: подойдут все типы, у которых базовый - int (и сам int)
```

**Вариант 2**. Интерфейс для всех сравниваемых типов (которые работают с операторами `==` и `!=`)

```go
[T comparable]
```

**Вариант 3**. Интерфейс для всех упорядоченных типов (которые работают со знаками неравенства `>`, `<`, `>=`, `<=`).

```go
[T cmp.Ordered]
```

**Вариант 4**. Любой тип (пустой интерфейс)

```go
[T interface{}]
// или:
[T any]
```

## Кастомные обобщенные типы

С помощью дженериков можно создавать кастомные типы:

```go
type Number interface {
	int64 | float64
}

type Numbers[T Number] []T // тип для слайса, эл-ты которого int64 или float64
```

Но при объявлении переменной с таким типом нужно указать конкретный тип:

```go
var numbers1 Numbers[int64] // конкретно задали, что эл-ты - int64
numbers2 := Numbers[float64]{1., 2., 3.} // а тут float64
```

Пример обобщенной структуры:

```go
type Pair[K, V any] struct {
    Key   K
    Value V
}

p1 := Pair[string, int]{Key: "age", Value: 30}
p2 := Pair[int, string]{Key: 30, Value: "age"}
```

> [!warning]
> Методы структур не могут содержать свои обобщенные параметры:
> ```go
> func (s SomeStruct) Method[T any](param T) T
> // ошибка: method must have no type parameters
> ```
> 
> Но если дженерик есть в определении структуры, то его можно использовать в методе (указываем рядом с ресивером):
> ```go
> type Container[T any] struct {
> 	value T
> }
> 
> func (c Contrainer[T]) Get() T {
> 	return c.value
> }
> ```


# Ошибки

## Ошибки и обработка

В Go нет привычных исключений, а есть ошибки. И ошибки в Go это специальные значения.
Обычно функции возвращают какое-то значение и ошибку (или только ошибку), и чтобы обработать ее, нужно использовать следующую конструкцию:

```go
_, err := fmt.Scan(&x)
if err != nil { // если ф-я вернула ошибку, а не nil
	fmt.Println(err) // то как-то ее обрабатываем
}
```

## Устройство ошибок

На самом деле ошибка - это интерфейс с методом Error:

```go
type error interface {
	Error() string
}
```

>[!important]
>Метод Error у типа ошибки принято реализовывать с получателем по указателю. Поэтому интерфейсу ошибки будет соответствовать именно указатель на тип. Пример из исходного кода Go:
>
>```go
>// errorString is a trivial implementation of error.
>type errorString struct {
>	s string
>}
>
>func (e *errorString) Error() string {
>	return e.s
>}
>```
>
>Поэтому обычно ошибка - это указатель на структуру. Функция `error.New`, например, возвращает `&errorString{s: <текст ошибки...>}`

## Работа с ошибками

#### `fmt.Errorf`, `errors.New`

Мы можем возвращать ошибку из функции, чтобы передать ее дальше в вызывающий код. Лучше всего возвращать ошибку, предварительно обернув ее в контекст (из которого потом мы сможем достать саму ошибку) с помощью `fmt.Errorf` и спецификатора `%w`:

```go
return fmt.Errorf("ошибка при обработке данных пользователя %s: %w", userID, err)
```

`fmt.Errorf` возвращает указатель на структуру `wrapError`:

```go
// из исходного кода Go:

type wrapError struct {
	msg string // наше сообщение
	err error // ошибка %w (указатель)
}

func (e *wrapError) Error() string {
	return e.msg
}

func (e *wrapError) Unwrap() error {
	return e.err
}
```

А с помощью метода `errors.New` можно создать значение ошибки:

```go
var ErrDatabaseTimeout = errors.New("Превышено время ожидания подключения")
err := ErrDatabaseTimeout

fmt.Printf("%T, %#v", err, err)
// *errors.errorString, &errors.errorString{s:"Превышено время ожидания подключения"} 

// добавляем обертку ошибки
wrappedErr := fmt.Errorf(
	"Ошибка при обработке данных пользователя %d: %w", 123, err,
)

fmt.Printf("%T, %#v", wrappedErr, wrappedErr)
// *fmt.wrapError, &fmt.wrapError{msg:"Ошибка при обработке данных пользователя 123: Превышено время ожидания подключения", err:(*errors.errorString)(0x17142fc1c030)}

// еще сообщение ошибки можно получить через Error()
fmt.Println(wrappedErr.Error())
// Ошибка при обработке данных пользователя 123: Превышено время ожидания подключения
```

#### `errors.Is`

Позволяет проверить, находится ли в обертке какая-то конкретная ошибка:

```go
if errors.Is(wrappedErr, ErrDatabaseTimeout) {
	fmt.Println("ErrDatabaseTimeout содержится в wrappedErr")
}
```

>[!tip]
>Под капотом работает так:
>- Вдруг мы передали не обертку, а сразу нужную ошибку? Сначала сравнивает `wrappedErr == ErrDatabaseTimeout`
>- Если верхний уровень не совпал, то вызывает у обертки метод `Unwrap` и получает значение поля `err` (это указатель)
>- Сравнивает этот указатель с указателем в `ErrDatabaseTimeout` и при совпадении возвращает `true`
>- Если цепочка оберток глубокая, то повторяет действия

#### `errors.Unwrap`

Используется, чтобы достать внутреннюю ошибку, которую мы поместили в обертку через `%w` (снимает ровно один слой обертки, т.е получает поле `err` у обертки):

```go
// относится к примерам выше

unwrapped := errors.Unwrap(wrappedErr)

fmt.Printf("%T, %#v", unwrapped, unwrapped)
// *errors.errorString, &errors.errorString{s:"Превышено время ожидания подключения"}

// проверка на равенство
if unwrapped == ErrDatabaseTimeout {
	fmt.Println("Совпало") // условие выполнится, вывод: Совпало
}
```

#### `errors.As`

Позволяет проверить содержится ли в обертке ошибка определенного типа и если содержится, то сохраняет ее в нашу переменную под ошибку.

```go
// свой тип ошибки
type MyError struct {
	Code int
	Msg string
}

// имплементим интерфейс error
func (e *MyError) Error() string {
	return fmt.Sprintf("[%d] %v", e.Code, e.Msg)
}

func main() {
	// имитируем получение обернутой ошибки
	originalErr := &MyError{404, "Страница не найдена"}
	wrappedErr := fmt.Errorf("Что-то пошло не так: %w", originalErr)
	
	var err *MyError // подготовили переменную под указатель нашей ошибки
	
	// важно: вторым аргументов передается указатель на переменную, чтобы записать в нее указатель на найденную ошибку
	if errors.As(wrappedErr, &err) {
		fmt.Printf("%T, %#v", err, err)
		// *main.MyError, &main.MyError{Code:404, Msg:"Страница не найдена"
		
		fmt.Println(err.Code, err.Msg) // 404 Страница не найдена
	}
}
```

>[!tip]
>Под капотом работает так:
>- Вызывает у обертки метод `Unwrap` и получает значение поля `err` (это указатель)
>- Сравнивает тип ошибки по указателю поля `err` с типом переменной, в которую хотим поместить найденную ошибку, если совпали, то записывает в переменную указатель из поля `err` и возвращает `true`
>- Если цепочка оберток глубокая, то повторяет действия



# Отложенный вызов defer

## defer

В Go можно отложить выполнение функции под конец работы родительской функции с помощью ключевого слова `defer`. При этом можно работать как уже с объявленными функциями, так и с анонимными.

>[!important]
>Внутри функции можно использовать много вызовов `defer`. Порядок их выполнения будет соответствовать LIFO: выполнение начнется с конца, последний `defer` выполнится первым и т.д

```go
func Example1() {
    for i := 1; i <= 3; i++ {
        defer fmt.Print(i, " ") // используем существующую функцию
        // порядок вывода: 3 2 1
    }
}

func Example2() {
    for i := 1; i <= 3; i++ {
        defer func(n int) { // анонимная функция
            fmt.Print(n, " ")
            // порядок вывода: 3 2 1
        }(i)
    }
}
```

>[!important]
>`defer` срабатывает даже после возникновения фатальных ошибок (паник)



# Паники

## Синтаксис

**Паники** - своего рода фатальные ошибки. После того, как в коде произошла паника, дальнейшее выполнение программы прекращается и мы получаем информацию о панике. Паника вызывается так: `panic(message)`

```go
func f() {
    panic("Something went wrong!")
}

func main() {
    fmt.Println("Start") // Start
    f() // panic: Something went wrong!
    fmt.Println("End") // эта строка не будет выполнена
}
```

>[!important]
>В Go действует принцип "Don't panic!", который говорит о том, что не нужно злоупотреблять вызовами `panic` в программе

## Обработка паник, `recover`

Можно обработать панику и продолжить выполнение кода с помощью `recover()`. Восстановление сработает только если `recover` вызвать внутри функции с `defer`. Если паника не случилась, то `recover` вернет `nil`:

```go
func Example() {
	fmt.Println("Start")
	
	defer func() {
		if r := recover(); r != nil {
			fmt.Println("Recovered:", r)
		}
	}()
	
	panic("Something went wrong!")
}

func main() {
	Example()
	// Start
	// Recovered: Something went wrong!
	
	fmt.Println("123") // 123
}
```
