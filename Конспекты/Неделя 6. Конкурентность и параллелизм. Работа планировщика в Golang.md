# Конкурентность и параллелизм. Работа планировщика в Go

## Конкурентная разработка

В разработке существует 2 важных понятия:
- **Параллелизм** - процесс выполнения нескольких задач одновременно в какой-то момент времени на разных ядрах процессора. Здесь задачи действительно выполняются параллельно.
- **Конкурентность** - процесс выполнения нескольких задач в рамках одного ядра таким образом, что выполнение постоянно переключается между задачами. Её профит в том, что мы можем распределить ресурсы одного ядра на несколько задач сразу, переключая выполнение с одной задачи на другую во время ожидания.

В Go можно реализовывать конкурентность с помощью **горутин**.

## Горутины, теория

На уровне ОС программы работают с помощью **потоков/тредов**, у них есть свои недостатки в виде фиксированного размера (максимум 8 Мб), ресурсоемкого создания (причем ограниченного по количеству - максимум тысячи тредов) и тяжелого переключения между одним и другим потоками.

По этим причинам в Go создали свою абстракцию - **горутины** - для выполнения задач поверх потоков ОС. **Горутина** - это единица выполнения конкурентной работы в языке.
Их преимущества:
- Динамический размер, который начинается всего с 2 Кб и может доходить до 1 Гб.
- Горутины создаются встроенными инструментами языка, а не через ОС, поэтому их создание - легкий процесс, их можно создать хоть миллион.
- Переключение между ними также осуществляется инструментами языка, поэтому этот процесс происходит быстрее.

Сами горутины представляют из себя обычные функции/методы, в которых код выполняется именно **последовательно** строка за строкой, а **планировщик Go** реализует конкурентность между горутинами, переключая одну на другую в моменты ожидания первой (чтение файла, обращение к базе). Кстати, в наших программах уже есть одна горутина по умолчанию - функция `main()`

## Работа с горутинами

Создание горутины очень простое - нужно указать ключевое слово `go` и после него вызвать необходимую функцию (можно и анонимную). Это дает планировщику команду создать горутину с этой функцией и запустить ее конкурентно.

```go
package main

import "fmt"

func sayHello(name string) {
	fmt.Printf("Привет, %s!\n", name)
}

func main() {
    fmt.Println("Начинаем программу...")
    
    // Запускаем первую горутину
    go sayHello("Анна")
    
    // Запускаем вторую горутину
    go sayHello("Борис")
    
    fmt.Println("Программа завершена")
}
```

Вывод у такого кода будет необычный:

```
Начинаем программу...
Программа завершена
```

А все потому что ключевое слово `go` лишь создает горутины, но они не успевают выполниться до завершения основной `main` горутины, после которой вся программа завершается. Если бы в `main` была какая-то задача с ожиданием, то горутины бы скорее всего успели выполниться, причем порядок их выполнения не гарантирован (он может быть рандомен).

## Работа планировщика, теория

Планировщик в Go работает по правилу M : N - некоторое количество горутин (M) выполняется на некотором количестве потоков ОС (N). Планировщик не управляет самими потоками ОС, но может решать, какие горутины и в каком порядке выполнить на определенном доступном потоке.

Модель работы планировщика - **GMP**:
- **Goroutine** - сама сущность горутины, содержащая код выполнения. У горутины могут быть разные статусы: пауза (в моменте ожидания), выполнение, готовность к выполнению (ожидание завершено).
- **Processor** - абстракция языка, которая соответствует ядру процессора. У нас может быть P <= количества ядер физического процессора. Каждый P содержит свою очередь выполнения горутин, в которую они попадают либо из глобальной очереди горутин **GRQ**, либо из соседних P (когда очередь на текущем P завершилась, то чтобы он не простаивал, половина свободных горутин перекидываются с другого P - **Work stealing**), либо если какая-то горутина из текущей очереди создает новую горутину. Очередь горутин на P ограничена - максимум 256 элементов.
- **Machine** - абстракция языка над реальным потоком ОС. Именно M выполняет горутины, для этого он прикрепляется к определенному P, а если в моменте горутина начала простаивать, то либо M переключается на другую горутину из P (если это легкое неблокирующее ожидание), либо блокируется ОС и ждет горутину (если это какой-то системный вызов, типо чтения файла) и тогда P продолжает выполнение других горутин из очереди уже с другим, свободным потоком M

В итоге концепцию можно описать так:
- **G** - задача
- **P** - где лежат задачи 
- **M** - кто их выполняет

#### Примеры сценариев работы планировщика

**Сценарий 1**. M не блокируется, перехватывает другую горутину:
1. Есть процессор P, у него в очереди находится 100 горутин G
2. К нему прикрепляется M и начинает выполнять горутины в порядке очереди: берет первую горутину
3. Если в какой-то момент эта горутина закончилась или переключилась в режим легкого ожидания, которое не связано с ОС, то M бросает ее выполнение и приступает к другой горутине из этого P

**Сценарий 2**. M блокируется операцией ОС, P переходит к другому M:
1. Есть процессор P, у него в очереди находится 100 горутин G
2. К нему прикрепляется M и начинает выполнять горутины в порядке очереди: берет первую горутину
3. Если в какой-то момент эта горутина переключилась в режим тяжелого ожидания, связанного например с чтением файла (задействуется ОС), то поток М блокируется и ожидает ее выполнение. Тогда планировщик открепляет P от этого М и ищет для него свободный М, чтобы продолжить выполнять горутины

> [!important]
> Мы можем управлять параллелизмом в Go - с помощью метода `runtime.GOMAXPROCS(n)` можно установить максимальное количество P, которые будут параллельно содержать в себе очереди горутин
> 
> ```go
> // Использовать только одно ядро — чистый конкурентный код
> runtime.GOMAXPROCS(1)
> 
> // Использовать все доступные ядра (по умолчанию)
> // Если ядер >2, то код и конкурентный, и параллельный одновременно
> runtime.GOMAXPROCS(runtime.NumCPU())
> ```

## Каналы

Синхронизация выполнения горутин осуществляется с помощью еще одного типа данных - **каналов** `chan`. Каналы организуют передачу данных между горутинами по принципу обмена сообщениями: горутина может получить сообщение от другой по каналу, локально поработать с этими данными и передать в другую горутину по каналу. Каналы работают в порядке очереди (FIFO - первое отправленное сообщение будет доставлено первым). Под капотом каналы представляют из себя структуры. 

Каналы создаются через функцию `make`: мы передаем в нее ключевое слово `chan` и тип данных который идет по каналу, а вторым необязательным параметром является размер буфера в случае, если канал должен быть буферизированным. Собственно каналы бывают двух видов.

**Вариант 1. `make(chan T)` - Небуферизированный канал**
Работает как прямая передача из рук в руки:
- **Блокировка при отправке в канал**: если горутина отправила какое-то сообщение по каналу, то она будет ожидать, пока принимающая горутина не освободится, чтобы забрать это сообщение
- **Блокировка при чтении из канала**: если горутина захотела получить данные из канала, то она будет ожидать, пока кто-то не отправит сообщение в канал

```go
func main() { // Горутина 1
    ch := make(chan int) // Небуферезированный канал
    
    go func() { // Горутина 2
        fmt.Println("Горутина 2: готова отправить")
        ch <- 42  // Блокируется пока горутина 1 не начнёт приём
        fmt.Println("Горутина 2: данные отправлены")
    }()
    
    time.Sleep(1 * time.Second) // Чтобы управление перешло во 2ю
    
    fmt.Println("Горутина 1: готова принять")
    value := <-ch  // Разблокирует горутину 2
    fmt.Printf("Горутина 1: получено значение %d\n", value)
    
    time.Sleep(1 * time.Second) // Чтобы управление перешло во 2ю
}
```

```
Горутина 2: готова отправить     # управление передалось во 2ю после sleep у 1й
Горутина 1: готова принять       # 2я ожидает получение, time.Sleep завершается,                                      управление снова у 1й
Горутина 1: получено значение 42
Горутина 2: данные отправлены    # управление передалось во 2ю после sleep у 1й
```


**Вариант 2. `make(chan T, n)` - Буферизированный канал**
Внутри него есть кольцевой буфер (хранилище). Это позволяет горутинам не ждать:
- **При отправке сообщения** горутина продолжит свое выполнение, если в буфере есть свободные места, иначе, если он полностью заполнен, горутина заблокируется и будет ожидать освобождения места под отправку сообщения.
- **При получении сообщени**я горутина получит его из буфера канала в порядке очереди (FIFO), а если буфер пустой, то будет ожидать первое отправленное сообщение.

```go
func main() {
    ch := make(chan int, 3) // Буфер на три элемента
    
    // Отправляем данные в буфер без блокировки
    fmt.Println("Отправляем в буфер:")
    ch <- 1
    fmt.Println("Отправлено 1, буфер: 1/3")
    ch <- 2
    fmt.Println("Отправлено 2, буфер: 2/3")
    ch <- 3
    fmt.Println("Отправлено 3, буфер: 3/3 (полон)")

    // Следующая отправка заблокирует буфер
    go func() {
        fmt.Println("Попытка отправить 4...")
        ch <- 4  // Заблокируется пока не освободится место
        fmt.Println("Отправлено 4")
    }()

    time.Sleep(500 * time.Millisecond)

    // Читаем из буфера
    value := <-ch  // Освобождает место для четвёртого
    fmt.Printf("Прочитано: %d\n", value)

    time.Sleep(500 * time.Millisecond)
}
```

```
Отправляем в буфер:
Отправлено 1, буфер: 1/3
Отправлено 2, буфер: 2/3
Отправлено 3, буфер: 3/3 (полон)
Попытка отправить 4...           # управление передалось после time.Sleep
Прочитано: 1
Отправлено 4
```

## Работа с каналами

Как видно из прошлых примеров работа с каналом осуществляется через `<-`:
- Для отправки сообщения в канал: `ch <- value`
- Для получения сообщения из канала: `value := <-ch`

Для получения есть привычная `value, ok := <-ch` нотация. Она нужна для проверки, закрыт канал или нет. Канал можно закрывать с помощью фукнции `close(ch)`, тогда передача данных по нему будет завершена.

> [!important]
> Канал должна закрывать только та горутина, которая в него записывает

>[!warning]
>Если не проверять закрытие канала при получении, то если канал действительно закрыт, то горутины продолжат получать дефолтные значения того типа данных под который открыт канал.
>
>Исключением является чтение канала в цикле `for range`, он автоматически проверяет его на закрытие:
>```go
>for val := range ch {
>	fmt.Println(val)
>}
>```

> [!warning]
> Запись в закрытые каналы или повторное закрытие канала - паника

## Однонаправленные каналы

Каналы могут быть как двунаправленными (и запись и чтение), так и однонаправленными (что-то одно). Однонаправленные задаются указанием типов:
- `<-chan` - только для чтения
- `chan<-` - только для записи

Причем двунаправленные всегда удовлетворяют типу однонаправленных каналов, но наоборот - нет:

```go
var bidirectional chan int = make(chan int)
var sendOnly chan<- int = bidirectional      // ✅ Разрешено
var receiveOnly <-chan int = bidirectional   // ✅ Разрешено
var bidir2 chan int = sendOnly               // ❌ Ошибка компиляции

// Пример использования однонаправленного канала
// Producer создаёт данные и отправляет их в канал (только отправка)
func producer(dataChan chan<- string) {}
```

## Работа с несколькими каналами

Мы можем ожидать получение данных сразу из нескольких каналов с помощью контрукции `select-case`. Горутина будет заблокирована `select`-ом, ожидая ответа от одного из своих кейсов. Причем если сразу несколько каналов готовы отдать сообщение, то `select` забирает даные в случайном порядке. Можно использовать оператор `default`, чтобы предотвратить блокировку `select` (если в моменте ни один из каналов не готов отдать данные)

```go
func main() {
	ch1 := make(chan string)
	ch2 := make(chan string)
	
	// Запускаем две быстрые горутины
	go func() { ch1 <- "Сообщение из канала 1" }()
	go func() { ch2 <- "Сообщение из канала 2" }()
	
	// Даем горутинам время дойти до каналов,
	// чтобы в момент select оба канала были готовыми
	time.Sleep(10 * time.Millisecond)
	
	select {
		case msg1 := <-ch1:
			fmt.Println("Победил:", msg1)
		case msg2 := <-ch2:
			fmt.Println("Победил:", msg2)
		default:
			// Сработает только если ни в одном канале нет данных
			fmt.Println("Данных нет, иду дальше")
	}
}
```

В этом примере обе горутины уже отправят сообщения в каналы, когда выполнение дойдет до `select`, поэтому вывод будет случайным их двух каналов

## nil-каналы

Канал может принимать значение `nil`, такой канал является неинициализированным. Все операции с ним будут блокироваться навсегда, а попытка закрыть такой канал вызовет панику.

## Гонки данных и состояний

**Race condition** - гонка состояния, возникает при одновременном доступе к данным и зависит от порядка выполнения горутин, которые в случае "одновременного" выполнения могут дать непредсказуемый результат, так как планировщик не гарантирует порядок выполнения

**Data race** - ситуация, когда несколько потоков одновременно обращаются к одним данным в памяти и хотя бы один поток изменяет эти данные. В таком случае другие потоки могут получить разные данные: либо до изменений, либо после.

```go
// ❌ Небезопасный инкремент
func simpleRaceCondition() {
    counter := 0
    
    go func() {
        counter = 1
    }()
    
    go func() {
    	counter = 100
    }()
    
    time.Sleep(1*time.Second)
    
    fmt.Printf("Результат: %d\n", counter)
    // Результат будет непредсказуемым: 1 или 100
}
```

#### Отслеживание гонок в Go

Go предоставляет встроенный детектор **race conditions** при запуске программы:

```bash
# Запуск программы с детектором
go run -race main.go

# Запуск тестов с детектором
go test -race ./...

# Сборка с детектором
go build -race main.go
```
